了解 GitHub Flow

5 分钟阅读 下载 PDF 版本

GitHub Flow 是一个轻量级的、以分支为基础的工作流，非常适合那些一直频繁的在部署系统的团队和项目。这篇指南将讲解 GitHub Flow 是如何工作的以及为什么它生效了。

创建一个分支

当你致力于一个项目时，你可能突然会有很多不同的建设性的功能或想法，一些可以马上动手实施，一些还不可以。分支的存在，将有效的帮助你管理这些不同想法的实施过程。

当你在你的项目中创建了一个分支，你就创建了一个可以让你尝试新想法的工作环境。你在分支中的变动不会影响到 master 分支，所以你可以任意的进行测试、做新版本等操作而不用担心被融合，除非你已经准备好跟你的队友审核这段代码。

专业建议

分支是 Git 中的一个核心概念，整个 GitHub Flow 都是以这个为基础的。它只有一条规则：任何 master 分支上的都一定是要发布实施的。

所以，当开发一个新功能或完成一个修复时，在 master 上创建一个新分支，都是极为重要的一步。你的分支名必须要很好的描述这个分支（例如, refactor-authentication, user-content-cache-key, make-retina-avatars），这样别人才能够知道这个分支是用来做什么的。

增加新版本

一旦你创建了新分支，你就可以开始做变更了。不论你何时增加、编辑或删除一个文件，你都是在做一个新的版本，你可以把他们加入到你的分支上。这个新增版本的过程，就跟踪了你在这个功能分支上的进展情况。

版本等同于创建了一个关于你工作情况的透彻的历史过程，这样别人就能跟随这个来了解你所做的以及为什么要这样做。每个版本都相应有一个用来叙述解释为什么要做这个改变的版本留言。此外，每个版本都被认为是一个单独的更新单位。这将允许你在发现bug时、或者你决定转向另一个方向时回滚这些变更。

专业建议

版本留言是非常非常重要的，特别是当 Git 开始跟踪你的变更并推送到服务器上作为版本显示时。通过书写一个清晰明了的版本留言，你可以让别人更容易跟随你的项目以及给你提供反馈。

开始一个 Pull Request

Pull Requests 用来发起一个关于你的版本的讨论。因为他们与底层 Git 仓库是紧密相联的，所以如果他们接受了你的 request ，任何人都可以完整的看到哪些变更将会被融合。

在开发过程中的任意时间点你都可以开始一个 Pull Request ：当你想到了一点点或者一点代码都没有，但你可以展示一些截图或大致想法时；当你遇到问题卡住了需要帮助或建议时；甚至当你准备好别人来审核你的工作时。通过在你的 Pull Request 信息中使用 GitHub 的“@mention”系统，你可以向特定的人或团队寻求反馈，而不用管他们现在是否近在眼前亦或者远在天边。

专业建议

Pull Requests 对于贡献开源项目以及管理公有共享仓库的更新是非常有用的。如果你正在使用 Fork & Pull 模式，Pull Requests 给你提供了一种方式来通知项目管理者，你希望他们考虑你做的变更。如果你正在使用一个 Shared Repository 模式（共享仓库模式），在这些变更被融合进 master 分支之前，Pull Requests 能帮助你们开始对所提出的变更进行讨论以及代码审查。

讨论、审核你的代码

一旦开启了一个 Pull Request ，审核你的变更的个人或团队可能会有问题或新的建议。可能代码风格不适合项目准则、变更缺少单元测试、又或者一切看起来很好并且井然有序。Pull Requests 就是被设计用于鼓励和引发这种类型的讨论。

你也可以根据对你版本的讨论和反馈，继续推送新代码到你的分支上。如果某人的评论指出了你忘做的事或者代码中的 bug ，你也可以在你的分支中修复它并推送这个变更。GitHub 将会在同一个 Pull Request 中显示你的新版本和任何你可能接纳的新的反馈。

专业建议

Pull Request 是用 Markdown 编写评论的，所以你可以插入图片和表情符号、用预格式化的文本块以及其他轻量级格式。

部署

一旦你的 pull request 被审核通过，并且分支通过了你们的测试，你就可以部署你的变更以便在实际产品上验证他们。如果你的分支引发了问题，你可以通过部署现有的主分支到产品上来实现回滚操作。

融合

现在你的变更已经在实际产品上验证过了，是时候把你的代码融合进主分支了。

一旦融合，Pull Requests 就会保存一个你代码的历史变更记录。因为他们是可搜索的，这就使得任何人都可以回顾了解当初为什么以及如何做这个决定的。

专业建议

通过结合你的 Pull Request 文本中的一些关键字，你可以用代码关联事务卡片。当你的 Pull Request 被融合了，这些相关的事务卡片也就被关闭了。例如：输入语句 Closes #32 ，就会关闭仓库中号码为32的事务卡片。更多信息，请参阅我们的帮助文档。
