Hello World 在电脑编程中是一个历史悠久的传统项目。当你在学习一些新东西时，它让你开始一个简单的练习。

GitHub 是一个在各个项目中用于托管和合作的平台。你不用担心你硬盘里的数据丢失，或者穿梭在多台电脑间管理一个项目 — sync(同步)从任何地方。最重要的是，GitHub 是一个用于更好在一起开发软件的协同异步工作流。

这个指南引领你穿越 GitHub 的各个要点：仓库(Repository)、分支(Branch)、版本(Commit)、事务卡片(Issue) 以及 Pull Requests。

拥有这些技能，你将会走的更远。通过这个 hello-world 仓库，你将有一个地方来贮存你的想法 (又名未来仓库，就像 jlord/hello-world)、资源或其他常规笔记，甚至和别人讨论一些事情 (就像 holman/feedback)。

正式使用 & 无需代码

我们将在本教程内用 GitHub.com 来完成这些任务，所以你不需要知道命令行甚至不需要安装 Git (就是那个 GitHub 所基于的变更追踪软件)。你甚至不需要知道如何编码。当然，如果你没有帐号的话，你需要注册一个！

小贴士: 在单独的浏览器窗口(或标签)打开本指南，这样你就可以同时看到你在本教程内完成这些步骤。

创建仓库

一个 仓库(Repository) 是 GitHub 的一个基础单元，常常也是一个单独的项目。仓库能放置文件夹和文件，包含图片 – 任何你项目需要的文件。在你创建新仓库的同时，由于我们推荐在每一个仓库包含一个 README ，或一个该项目的描述文件，所以你可以很容易的添加一个。 GitHub 也提供其他常规设置，例如一个许可文件，但我们现在可以跳过这个步骤了

创建一个新仓库

  1. 点击 图标，在你的用户名后面，顶部右侧。
  2. 命名你的仓库 hello-world。
  3. 写一个简短的描述( Description )。
  4. 选择 Initialize this repository with a README.

点击 Create repository. 砰~叭~, repository!

开启事务卡片

一个 事务卡片(Issue) 是一个仓库的便签，涉及一些需要关注的东西。它可能是个 bug 、是个功能请求、是个问题、甚至其他许多东西。在 GitHub 上，你可以标记、搜索、分类事务卡片，更容易的管理一个进行中的项目。

你现在已经有了一个仓库了，但还颇空。可以在 README 里添加更多的信息，这样别人就知道这个仓库的用途了。来开启一个事务卡片吧！

为编辑 README 开启一个事务卡片

  1. 在侧边栏点击 Issues。
  2. 点击 New Issue。
  3. 给你的 Issue 一个标题和描述。

当你完成以上步骤，点击 Submit new Issue。现在这个事务卡片有一个自己的永久主页了，哪怕在它关闭之后你也可以引用它。

接下来，开始编辑你的 README ，然后关闭这个事务卡片。

创建分支

分支(Branching) 是一种在同一时间同一仓库内编辑不同部分的方法。

当你创建了一个新仓库，默认会生成一个名为 master 的分支。你将在这唯一的分支上进行开发，那真不错。但如果你想开发一个其他的功能或新的想法，你可以创建另一个分支，复制于 master ，这样你就可以把 master 抽离工作状态了。

当你创建一个分支同时，你就从原分支上做了一份拷贝(就像相片快照)。如果原分支有更新，而你正致力于新分支时，也不用担心，你仍然可以随时推送这些更新。

在你的生涯中，你可能保存了很多不同的版本文件，例如：“story.txt”, “story-joe-edit.txt”, “story-sue-edit.txt”。分支也是为了完成这同一个目的，不过它更易于在 GitHub 仓库上管理。

在 GitHub 上，我们的开发者、写作者、设计者用分支来独立于 master (production) 不断进行bug的修复和功能的开发。当功能完成、 bug 修复完成，这个分支就可以融合进 master 分支。

创建一个新分支

  1. 进入你的新仓库 hello-world。
  2. 点击文件列表顶部名为 branch: master 的下拉菜单。
  3. 在新分支文本框内，输入一个分支名, readme-edits。
  4. 选择蓝色的 Create branch 框，或敲击键盘上的 “Enter” 键。

现在你有两个分支了，master 和 readme-edits。他们看起来完全一样，但不会太久！接下来我们将在新分支上新增我们的更新。

生成版本

在 GitHub 上，保存的变更叫做 commits(版本) 。版本是非常有用的，因为这一连串的版本读起来就像是你项目的历史。

每个版本都有一个相关的 commit message (版本留言) ，详细描述解释了为什么要做这个变更。多亏这些留言，你和其他人可以通读这些版本并了解你做了哪些变更以及为什么要做这些变更。

你还是得根据代码查看你的 readme-edits 分支，现在来做些变更吧！

提交更新

  1. 点击 README file。
  2. 点击文件视图的右上角铅笔图标进行编辑。
  3. 在编辑器内，写一些文本，告知一些关于你自己的事情。
  4. 写一个描述你的更新的版本留言。

点击 Commit changes 。现在这些更新已经被做进你的 readme-edits 分支的 README 文件中了，现在这个分支包含了与 master 分支不同的内容和版本(正如预料的那样！)。

开启 Pull Request

Pull Requests 是 GitHub 上协作的核心。当你开启一个 pull request ，你就提交了你的变更并且希望他们接受你的贡献 - 又名融合进他们的分支。 GitHub 的 Pull Request 功能允许你对比两个分支上的内容。这些更新、新增以及删减显示为绿色和红色，称作 diffs （差异）。

一旦你做了变更，你可以开启一个 Pull Request 。人们甚至常用 Pull Request 在代码完成前开始一个关于版本(代码审查)的讨论。通过这个方法你就可以获得相应的反馈，而不论是你干劲十足时又或者是卡住了需要帮助时。

通过在你的 Pull Request 留言中使用 GitHub 的 @mention 系统，你可以向特定的人或团体寻求反馈，而不论他们远在天边抑或近在眼前。

当我们在这里一同工作时，你甚至也可以在你自己的仓库开启一个 Pull Request 然后自己融合它。在致力于大项目前，这是一个很好的学习做事方法的途径。

为 README 的变更创建一个 Pull Request

跟随图片点击来创建一个大版本

步骤 屏幕动图
在侧边栏点击 Pull Request 图标，然后在 Pull Request 页面，点击绿色的 New pull request 按钮。
选择你创建的分支， readme-edits ，与 master (原始分支)做对比。
在对比页面的差异上查看你的变更，确定它们就是你想要提交的。
当你确信这些就是你想要提交的变更时，点击这个大大的绿色的 Create Pull Request 按钮。
给你的 pull request 一个标题，因为它直接关系到一个公开的事务卡片，包含 “fixes #” 以及标题中的事务卡片的号码。为你的变更写一个简短的描述。

当你写完了你的留言，点击 Create pull request!

小贴士: 你可以在版本和 Pull Requests 中用 emoji 以及拖放图片和 gifs 动图

融合你的 Pull Request

是时候把你的变更集合起来了 - 把 readme-edits 分支融合进 master 分支。

  1. 点击那个绿色的按钮来把更新融合进 master 。
  2. 点击 Confirm merge.
  3. 继续往下通过紫色框内的 Delete branch 按钮来删除这个分支，因为它的更新已经被融合了。

如果你再去查看你之前开启的事务卡片，它现在已经关闭了！因为你在你的 Pull Request 标题中包含了 “fixes #1” ，当 Pull Request 被融合的时候，GitHub 负责关闭该事务卡片！

祝贺你!

你已经学会了在 GitHub 上创建一个 pull request !
